## Current Blocker: Duplicate `api_name` Conflicts

The duplicate guard in `_build_lookup_maps()` correctly identified that multiple `usda_commodity` rows share the same `api_name`, which breaks the 1:1 mapping assumption.

### Confirmed Conflicts

**GRAPES** — Two `usda_commodity` rows both have `api_name = 'GRAPES'`:

| commodity_name | usda_code | api_name | Resources mapped to it |
|----------------|-----------|----------|----------------------|
| `ALL GRAPES` | 21699999 | `GRAPES` | grapes, grape stem, grape vines |
| `WINE GRAPES` | 21629999 | `GRAPES` | grape pomace, diatomaceous earth from winemaking, lees, spent oak chips, spent oak stick, spent yeast, unused oak stick |

**WHEAT** — Two `usda_commodity` rows both have `api_name = 'WHEAT'`:

| commodity_name | usda_code | api_name | Resources mapped to it |
|----------------|-----------|----------|----------------------|
| `WHEAT` | 10199999 | `WHEAT` | wheat hay, wheat straw |
| `WHEAT WINTER ALL` | 10119999 | `WHEAT` | winter wheat |

### Root Cause

In `reviewed_api_mappings.py`, multiple NASS commodity code names map to the same QuickStats `commodity_desc`:
- `'ALL GRAPES': 'GRAPES'` and `'WINE GRAPES': 'GRAPES'`
- `'WHEAT': 'WHEAT'` and `'WHEAT WINTER ALL': 'WHEAT'` (plus 9 other wheat variants)

This is correct for the *API query* (QuickStats uses `GRAPES` for all grape data), but when these are seeded as separate `usda_commodity` rows in the DB, the `api_name` column is no longer unique, and the transform's `commodity_map` dict can only hold one ID per `api_name`.

### API Verification

Queried the QuickStats API directly to check available `commodity_desc` values:

```
https://quickstats.nass.usda.gov/api/get_param_values?param=commodity_desc
```

Results for grape/wheat related entries:
```
BUCKWHEAT
GRAPEFRUIT
GRAPES
WHEAT
```

**Confirmed:** QuickStats has only `GRAPES` (no `WINE GRAPES`) and only `WHEAT` (no `WHEAT WINTER ALL`). These sub-categories are NASS county data file codes, not QuickStats API `commodity_desc` values.

### Decision: Option B — Consolidate in CSV

**Option A (distinct api_names)** was ruled out because `api_name` is used directly as the `commodity_desc` parameter in the API query. Values like `GRAPES - WINE` would return nothing from the API.

**Option B (consolidate)** was chosen:
- Changed 7 rows in `commodity_mappings.csv`: `WINE GRAPES` (21629999) → `ALL GRAPES` (21699999)
- Changed 1 row: `WHEAT WINTER ALL` (10119999) → `WHEAT` (10199999)
- `reviewed_api_mappings.py` left unchanged — the `'WINE GRAPES': 'GRAPES'` mapping is still valid as a safety net for `get_api_name()` lookups
